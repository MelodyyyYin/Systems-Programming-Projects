# Cache Lab 面试展示提纲

下面这份笔记是给我自己讲项目用的，不是提交文件。目标是让面试时可以用 3 到 5 分钟把这个 lab 讲清楚，后面再按追问展开。

## 1. 一句话概括

这个 Cache Lab 主要做两件事：

1. 写一个 cache simulator，按给定的 `s/E/b` 参数模拟访问，统计 hits、misses、evictions 和 dirty bytes。
2. 写一个高性能矩阵转置，利用 cache 局部性降低 miss 数，在指定矩阵规模下尽量接近评分上限。

如果面试官只想先听结论，我会先说：

- Part A 重点是“正确建模 cache 行为”。
- Part B 重点是“针对不同矩阵尺寸设计不同的访问模式”。

## 2. 我怎么介绍 Part A: `csim.c`

### 2.1 核心设计

我把 cache simulator 拆成了三层：

- 参数解析层：处理 `-h -v -s -E -b -t`，并检查缺参、非法参数、额外参数。
- trace 解析层：逐行读取 trace，严格校验格式，避免把脏输入当成合法访问。
- cache 模拟层：按 set / line 组织数据，使用 LRU 选择替换行，同时记录 dirty 状态。

### 2.2 关键数据结构

我会强调这几个字段：

- `valid`: 这行是否有效。
- `tag`: 用于命中判断。
- `dirty`: 写操作后是否被修改。
- `lru_ts`: 最近使用时间戳，用来做 LRU。

这几个字段足够覆盖 lab 要求，不需要上更复杂的数据结构。

### 2.3 模拟逻辑

一次访问分三步：

1. 先在目标 set 里找 tag 是否命中。
2. 命中则更新 hit 计数和 LRU 时间戳，写操作会把 line 标成 dirty。
3. 未命中则计 miss，优先找空位；如果没有空位，就按最旧的 `lru_ts` 淘汰。

如果被淘汰的 line 是 dirty，就把被写回的 block 大小计入 `dirty_evictions`。
最后再扫描 cache，统计仍然留在 cache 里的 dirty bytes。

### 2.4 我会怎么说明正确性

我会提两点：

- 结果必须和 `csim-ref` 对齐，所以我不仅看 hits/misses，也看 evictions 和 dirty 相关统计。
- trace 解析是严格的，避免因为“宽松解析”把错误输入默默吞掉，导致和 reference 行为不一致。

## 3. 我怎么介绍 Part B: `trans.c`

### 3.1 总体思路

转置的瓶颈不是算术，而是内存访问模式。我的策略是按矩阵尺寸分情况处理：

- `32x32`：使用 `8x8` 分块，减少冲突 miss。
- `1024x1024`：同样使用 `8x8` 分块，但对对角块和非对角块区别处理。
- 其他尺寸：回退到正确性优先的普通转置，保证所有输入都能通过验证。

### 3.2 为什么要分块

转置里最糟糕的情况是按列写 `B`，会破坏 cache 局部性。分块以后：

- 读取 `A` 时更连续。
- 写入 `B` 时也更集中。
- 通过把工作集限制在小块里，可以减少 cache 冲突。

### 3.3 32x32 的做法

对 `32x32`，我用 `8x8` 块直接搬运，并把对角元素单独延后处理，避免自己读写同一块时产生额外冲突。

如果面试官追问“为什么对角元素要特殊处理”，我会回答：

- 对角块里 `A[i][j]` 和 `B[j][i]` 可能落在相近的 cache 区域。
- 先处理非对角元素，再处理对角元素，可以降低不必要的覆盖和冲突。

### 3.4 1024x1024 的做法

这个规模更大，所以我还是按 `8x8` 处理，但在对角块里借助临时数组 `tmp` 做中转：

- 先把块读进 `tmp`。
- 再从 `tmp` 写到 `B`。

这样做的目的是把某些高冲突访问变成更可控的顺序访问。

### 3.5 我会怎么解释“为什么不是所有情况都极致优化”

因为 lab 评分是分规模的，最重要的是在目标规模上拿到高分，同时保证所有输入都正确。

- 复杂优化通常只对少数尺寸有收益。
- 过度特化会增加 bug 风险。
- 所以我保留了清晰的 fallback 路径，优先保证正确性，再做针对性优化。

## 4. 面试时可以主动讲的亮点

- 我不是只写功能，而是把“正确性”和“性能”分开考虑。
- `csim` 里我显式处理了 dirty cache line 和 write-back，这比只统计 hit/miss 更完整。
- `transpose` 里我根据矩阵尺寸选择不同策略，体现了“针对 cache 行为做优化”的思路。
- 我保留了测试入口：`test-csim`、`test-trans`、`driver.py`，方便验证实现。

## 5. 可被追问的问题

1. 为什么 LRU 可以用时间戳实现？
2. write-allocate 和 write-back 在这里分别体现在哪里？
3. 为什么 8x8 分块比逐元素转置更适合这个 lab？
4. 对角块为什么更容易产生冲突 miss？
5. 你是怎么确保 `A` 不被修改的？

## 6. 我自己的讲述顺序

如果时间很短，我会按这个顺序讲：

1. 先说 lab 目标。
2. 再说 `csim` 的 cache 模型和统计项。
3. 再说 `trans` 的分块策略。
4. 最后说测试方式和结果。

这样面试官能很快判断我是不是理解了 cache 行为，而不只是把代码写出来了。
