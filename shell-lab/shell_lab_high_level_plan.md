## 1. 项目目标与范围

本项目目标是基于 [`tshlab_technical_spec.md`](/Users/yueyin/Desktop/18613-sp26/shell-lab-MelodyyyYin/tshlab_technical_spec.md) 完成一个可提交的 `tsh` 小型 Linux shell，实现与参考实现 `tshref` 一致的行为，并通过官方 trace/driver 验证。

范围内工作包括：

- 完成 shell 主循环与命令执行流程。
- 支持前台与后台作业。
- 支持作业控制、PID/JID 寻址与作业状态管理。
- 支持内建命令 `quit`、`jobs`、`bg`、`fg`。
- 支持 `SIGCHLD`、`SIGINT`、`SIGTSTP` 相关行为。
- 支持外部命令的输入输出重定向，以及 `jobs` 的输出重定向。
- 满足竞态安全、输出格式一致性、错误处理和提交要求。

明确不在范围内的内容：

- 管道。
- 终端控制 API，如 `tcsetpgrp`、`tcsetattr`。
- 依赖 `sleep`、轮询或其他仅为通过 trace 的规避性做法。

## 2. 必须支持的功能

项目必须覆盖以下能力：

- Shell 交互入口：打印提示符、读取输入、在 EOF 时退出。
- 外部命令执行：正确传递参数与环境，支持前台和后台模式。
- 作业模型：维护 PID、JID、前台/后台/停止三类状态，并保持与进程生命周期一致。
- 内建命令：
  - `quit`：退出 shell。
  - `jobs`：列出后台作业，并支持 `jobs > file`。
  - `bg <job>`：按 PID 或 `%JID` 恢复停止作业并转入后台。
  - `fg <job>`：按 PID 或 `%JID` 恢复作业并转入前台。
- 信号处理：
  - `SIGCHLD`：及时回收子进程，处理退出、被信号终止、被停止三类状态变化。
  - `SIGINT`、`SIGTSTP`：仅在存在前台作业时转发到整个前台进程组。
- 进程组隔离：外部命令不能与 shell 共享前台进程组语义。
- 重定向：
  - 外部命令支持 `<`、`>`、同时输入输出重定向，以及不同顺序的重定向写法。
  - `jobs` 支持输出重定向。
- 错误处理：
  - routine 用户错误不能导致 shell 异常退出。
  - 输出行为与 `tshref` 一致，允许的差异仅限技术文档说明的 PID 和特定 `ps` 输出差异。

## 3. 分阶段开发计划

### 阶段 0：基础启动与退出

目标：

- 建立 shell 主循环的最小可运行路径。
- 支持 EOF 退出与 `quit`。

完成标志：

- `./tsh` 可启动。
- `trace00`、`trace01` 通过。

依赖关系：

- 无前置实现依赖，是后续所有能力的基础。

### 阶段 1：前台命令执行

目标：

- 支持非内建命令的基本执行。
- 保证参数与环境透传正确。
- 建立前台命令完成前 shell 的基本等待语义。

完成标志：

- 前台外部命令可正常运行。
- `trace02` 到 `trace04` 通过。

依赖关系：

- 依赖阶段 0 的命令循环与内建命令分流。

### 阶段 2：后台作业与基础作业管理

目标：

- 支持 `&` 启动后台作业。
- 建立作业表、JID 分配和后台启动输出。
- 将子进程状态回收统一纳入 `SIGCHLD` 路径。
- 从这一阶段开始统一建立基于 `SIGCHLD` 的回收模型，不再保留分散的前台回收路径。

完成标志：

- 后台作业启动后提示符立即返回。
- `jobs` 能列出后台作业。
- `trace05` 到 `trace08` 通过。

依赖关系：

- 依赖阶段 1 的外部命令执行能力。
- 为后续信号处理和 `bg`/`fg` 提供作业状态基础。

### 阶段 3：信号处理与进程组行为

目标：

- 在阶段 2 已建立的 `SIGCHLD` 回收路径之上，补齐 `SIGINT`、`SIGTSTP` 转发语义。
- 建立前台进程组的信号转发规则。
- 确保作业状态变化、消息输出、回收时序与竞态安全一致。

完成标志：

- 前台作业可被正确中断和停止。
- shell 自身不会错误接收前台作业信号语义。
- `trace09`、`trace11` 到 `trace21` 通过。

依赖关系：

- 依赖阶段 2 的作业表和后台/前台基本区分。
- 是 `bg`/`fg` 正确行为的前提。

### 阶段 4：`bg` / `fg` 作业控制

目标：

- 支持通过 PID 或 `%JID` 定位作业。
- 支持停止作业恢复、前后台状态切换、前台等待恢复。
- 对非法参数、缺失参数和竞态场景给出与 `tshref` 一致的行为。

完成标志：

- `bg` / `fg` 可驱动完整作业状态流转。
- `trace22` 到 `trace27` 通过。

依赖关系：

- 依赖阶段 3 的信号处理、停止/继续语义和前台等待规则。

### 阶段 5：重定向、错误处理与提交收尾

目标：

- 补齐 `<`、`>`、组合重定向和顺序差异。
- 完成 `jobs > file`。
- 收敛权限错误、文件错误和最终用户可见输出。
- 完成提交产物准备。

完成标志：

- `trace28` 到 `trace32` 通过。
- `make` 成功，且可生成提交产物。

依赖关系：

- 依赖前面各阶段的命令执行、内建命令和作业控制框架。

## 4. 测试与验证策略

测试应按“手工冒烟验证 -> 单 trace 定点调试 -> 全量 driver 回归”的顺序推进。

核心策略如下：

- 每完成一个阶段，先做最小手工验证，确认 shell 仍可交互、退出、执行基本命令。
- 按技术文档给出的 trace 分组推进，不跨阶段同时处理大量问题。
- 对语义不确定或输出格式不确定的场景，以 `tshref` 为唯一行为对齐基准。
- 使用 `runtrace` 做单 trace 调试，优先定位阶段内失败点。
- 使用 `sdriver` 做回归验证，并关注重复运行下是否暴露竞态问题。
- 将“通过一次”视为不充分，必须验证重复运行下结果稳定。
- 不将 `trace10` 计入通过目标，应以其余有效 traces 为回归集合。

重点验证维度：

- 命令生命周期：启动、退出、EOF、前台执行、后台执行。
- 作业生命周期：创建、列出、停止、恢复、转前台、转后台、删除。
- 信号语义：只作用于前台作业、作用于整个进程组、状态消息正确。
- 重定向语义：输入、输出、组合、顺序变化、权限或文件错误。
- 输出一致性：消息内容和时机与 `tshref` 对齐。
- 竞态鲁棒性：快速退出、多子进程同时变化、信号与作业表并发访问。

输出对齐检查项应至少覆盖：

- 后台作业启动消息格式。
- 作业因信号终止或停止时的消息格式。
- `bg` / `fg` 参数缺失、格式错误或目标无效时的提示。
- `jobs > file` 的输出与重定向行为。

## 5. 风险与边界情况

项目的主要风险点如下：

- 作业表竞态：子进程过快退出，导致父进程登记和回收路径交错。
- 双重等待：前台等待路径与 `SIGCHLD` 路径重复处理子进程状态，造成逻辑冲突。
- 进程组处理错误：前台信号影响 shell 本身，或只影响单个进程而非整个作业。
- 停止状态遗漏：只处理退出，不处理被停止的子进程，导致 `bg` / `fg` 语义失真。
- 多事件合并：一次 `SIGCHLD` 对应多个状态变化，如果只处理一次会遗留僵尸进程或错误状态。
- handler 输出不安全：在信号处理上下文中使用不安全输出方式。
- 重定向影响 shell 自身标准输入输出，导致后续交互异常。
- `bg` / `fg` 参数解析错误：PID 与 `%JID` 混淆，或缺失参数时行为不符。
- 错误处理不一致：普通用户错误导致 shell 退出，或报错文本与参考行为不一致。

需要特别关注的边界情况：

- 空输入行与 EOF。
- 没有前台作业时接收 `SIGINT` 或 `SIGTSTP`。
- 后台作业启动后立即退出。
- 单次 `SIGCHLD` 对应多个子进程状态变化。
- `bg` / `fg` 作用目标在解析或执行时已经发生状态变化。
- 重定向文件不存在、不可读、不可写或权限不足。
- `jobs` 的输出重定向与普通外部命令重定向的差异。

## 6. 最终交付物与完成标准

最终交付物应包括：

- 一份可构建、可运行的 `tsh` 实现。
- 满足技术文档要求的 shell 行为和用户输出。
- 可用于提交的打包产物或提交命令结果。

完成标准如下：

- `make` 在指定环境中成功。
- `./tsh` 可正常启动并接受命令。
- `./sdriver` 通过所有有效 traces，且不将已知 buggy/removed 的 `trace10` 作为目标。
- 输出行为与 `tshref` 一致，且仅存在技术文档允许的差异。
- 实现不依赖忙等、轮询、`sleep` 规避或超出范围的终端控制接口。
- 子进程状态回收路径保持集中，不在多个位置分散竞争 `waitpid`。
- signal handler 满足信号安全约束，包括保存/恢复 `errno`，并仅使用 async-signal-safe 的输出方式。
- 前台等待机制不采用 busy wait，而应符合技术文档要求的信号驱动等待语义。
- 代码结构满足可维护性要求，便于后续 agents 继续按阶段推进或补充修正。
